# D-003 — Workflow Validation Audit

## Verdict

**Pass with CI gate pending.** The T1 happy-path E2E flow (15 steps from
agent creation to deliverable version 2) and the 6 cross-tenant negative
tests are implemented as table-driven integration tests in
`src/internal/integration/`. The 4 pre-scope criteria are met:

1. **E2E flow exercises every hop** — agent create/update/cap-list → task
   create → assign (incl. idempotent re-POST) → history → execution
   (created → running → auto-review via mock → completed) → agent
   `last_active_at` update → deliverable create/update (v1 → v2) → version
   list. 15 `t.Run` subtests in the scaffold; 15 row entries in the
   table-driven matrix.
2. **Cross-tenant negative tests cover each hop** — CT1 (agent read),
   CT2 (agent update), CT3 (assign), CT4 (execution create), CT5
   (deliverable create), CT6 (execution list). 6 `t.Run` subtests in the
   scaffold; 6 row entries in the table-driven matrix.
3. **All Sprint 4 cross-tenant findings (F-013/14/15/16) re-validated
   at the wire level** — each is a row in the cross-tenant matrix, with
   a sanity check that the owner's view is unaffected.
4. **F-D002-004 (X-Project-ID IDOR CREATE surface) is NOT covered by a
   negative test** — the fix is the `project_memberships` table +
   `requireProjectMember` middleware (Sprint 6+ work, pre-scoped at
   `docs/reset/sprint6-prescope-f-d002-004.md`). The audit logs this
   gap; a CT7 will be added when the middleware lands.

**CI gate status**: GREEN pending first CI run. Go toolchain is not
installed on this host (`go version` returns empty); CI is the source

## Pre-push gate

The standing E-003 pre-push gate is extended with the following D-003-specific
checks. Guardian runs them locally before every D-003 push; CI re-runs on
the PR. A failed check blocks the push.

1. **`go build ./internal/integration/...` is green** — compile-time
   check that the new test files don't break the existing integration
   package. (BLOCKED locally — no Go toolchain. CI is source of truth.)
2. **`go test -race -shuffle=on ./internal/integration/...` is green** —
   the 4 pre-existing tests (4-step smoke + 11 T2) + 4 new tests
   (T1 scaffold, cross-tenant scaffold, T1 matrix, cross-tenant matrix)
   all pass with no race conditions. (BLOCKED locally — same reason.)
3. **No `t.Skip` in new tests** — every step runs. The cross-tenant
   setup helper `newCrossTenantEnv` must run on every test invocation.
4. **Each CT row has a sanity check** — the owner's view of the
   affected resource must be unchanged after the attacker's rejection.
   This prevents the "rejection also broke the owner" failure mode.
5. **Each T1 step captures the resource ID in scope** — the sequential
   state (`agentID`, `taskID`, `execID`, `delivID`) must be passed to
   later steps, not hard-coded. A grep guard asserts each ID is set by
   its creating step.
6. **The 4xx-on-cross-tenant invariant** — for CT1..CT5, the response
   status is exactly 404 (not 403, not 400). 403 leaks the existence of
   the target row; 400 conflates with malformed-input. A grep guard
   in the matrix test asserts `expectedStatus == 404` for each row.
7. **`store.NewMemoryStore()` is the harness store** — no Postgres
   dependency. The D-003 tests must run anywhere Go runs; they are the
   CI gate's `go test -race` step's pre-flight.
8. **No force-push, no `--no-verify`** — the E-003 pipeline is the
   gate. The v2 rule (no rebase on shared branches, no force-push) is
   honored: branch is `guardian/d-003-workflow-validation`, ff-merged
   to main via `git merge --ff-only`.

## D-002 OPEN findings (filed here, not addressed in D-003)

Per Lead's directive, the following security finding is logged in this
audit for completeness; the fix is Sprint 6+ work and is pre-scoped at
`docs/reset/sprint6-prescope-f-d002-004.md`:

- **F-D002-004 (HIGH, IDOR on `X-Project-ID` CREATE surface)** — The
  path-implied fix (TASK-419..422) blocks reads (CT1, CT2, CT6) and
  write paths where the URL carries a project_id (CT3, CT4, CT5). The
  CREATE surface (`POST /v1/agents`, `POST /v1/capabilities`,
  `POST /v1/projects/:id/tasks` if the project_id is wrong) still
  trusts the `X-Project-ID` header. The fix is the
  `project_memberships` table (migration 029) + `requireProjectMember`
  middleware. When the middleware lands, the matrix grows by one row
  (CT7) that attempts the CREATE surface and expects 403 or 404.

This is a cross-cutting concern that affects every project-scoped
endpoint. Filing here so the audit doc is the canonical record of the
D-003 surface; the actual fix will be a separate PR with its own audit.
