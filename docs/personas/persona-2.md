# Persona 2: Sarah Kim — Product Manager

## Demographics & Background
- **Name:** Sarah Kim
- **Role:** Senior Product Manager
- **Company:** CloudScale (B2B SaaS, 85 employees)
- **Age:** 29
- **Location:** Austin, TX
- **Education:** MS Human-Computer Interaction from Carnegie Mellon
- **Experience:** 5 years in product management, previously at Atlassian and Figma

## Goals & Motivations
- Define clear requirements that engineering can execute against
- Reduce the gap between "what users want" and "what gets built"
- Ship features faster without sacrificing user experience
- Have data-driven confidence in prioritization decisions
- Free up time for user research instead of writing tickets

## Pain Points
1. **Requirements ambiguity** — Engineers interpret specs differently, leading to rework
2. **Translation overhead** — Spends hours converting user feedback into technical tickets
3. **Scope negotiation** — Constant back-and-forth with engineering about what's feasible
4. **Context switching** — Juggling 3 projects simultaneously means nothing gets deep attention
5. **Stakeholder misalignment** — Different departments have different priorities

## How Sarah Interacts with the Platform
- Writes user stories and acceptance criteria in the platform
- Reviews the PM Agent's decomposition of features into tasks
- Validates that the Architect Agent's designs match user needs
- Monitors QA results to ensure acceptance criteria are met
- Uses analytics to justify prioritization to leadership

## A Day in the Life

**8:30 AM** — Opens the platform. The PM Agent has already broken down yesterday's feature request into 12 user stories with acceptance criteria. Sarah reviews and adjusts priorities.

**9:15 AM** — User research session. Notes key insights: "Users want bulk import." Enters this as a new project. The PM Agent generates a feature spec within minutes.

**10:00 AM** — Sprint planning meeting. Uses the platform's backlog view to show the team what's coming. The AI-generated stories are well-structured — engineers ask fewer clarifying questions.

**11:00 AM** — Reviews the Architect Agent's API design for the new feature. Notices the pagination approach doesn't match user expectations. Leaves a comment; the Architect Agent revises.

**1:00 PM** — Checks QA results. 87% of acceptance criteria pass. Reviews the failing tests — one is a genuine bug, one is a missing acceptance criterion. Updates the story.

**2:30 PM** — Writes a PRD for Q3 features. The platform's analytics show which features drive the most engagement. Uses this data to prioritize.

**4:00 PM** — Stakeholder sync. Shows the product roadmap with real progress data. "We're on track for the July release because the AI agents handled the backend work in parallel."

**5:30 PM** — Reviews the day's completions. Three stories moved to "Done." The Review Agent flagged two code quality issues that were already fixed.

## Success Criteria for Sarah
- Requirements are clear enough that engineers don't need to ask questions
- User stories have measurable acceptance criteria
- She spends 60% less time writing tickets and 60% more time on user research
- Every feature decision is backed by data
- Cross-team alignment happens through shared visibility, not meetings
